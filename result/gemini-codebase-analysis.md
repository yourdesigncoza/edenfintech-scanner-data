# EdenFinTech Scanner — Codebase Analysis (Gemini)

## 1. Overall Assessment

This is a **highly sophisticated, agentic workflow** designed specifically for the Claude Code environment. It is not merely a script, but a **multi-agent system** that enforces a specific investment philosophy (Deep Value Turnaround).

The architecture follows a "Manager-Worker" topology:
1. **Orchestrator:** Manages state, enforces portfolio rules, and compiles reports.
2. **Screener:** High-volume, low-compute filtering (Quantitative).
3. **Analyst:** Low-volume, high-compute reasoning (Qualitative + Valuation).
4. **Tooling Layer (`fmp-api.sh`):** A robust backend that handles data fetching, caching, and rate limiting.

The code is well-structured, modular, and exhibits "Prompt Engineering" best practices (e.g., role definition, few-shot prompting, chain-of-thought requirements, and explicit constraints). It effectively bridges the gap between raw API data and high-level investment reasoning.

---

## 2. Strengths

### Rigorous Caching Strategy
The `fmp-api.sh` script implements a file-based caching system with specific TTLs (Time To Live) for different endpoints (e.g., prices expire in 1 day, balance sheets in 90 days). This is critical for cost control and speed.

### Separation of Concerns
The Screener and Analyst have distinct "personalities" and instructions. The Screener is told to be "ruthless" and "fast," while the Analyst is told to be "skeptical" and "detailed."

### Psychological Guardrails
The prompts explicitly try to counter LLM hallucinations and sycophancy.
- *Example:* "Finding ZERO recommendable stocks is a valid and valuable outcome."
- *Example:* The "Heroic Assumptions Test" forces the model to flag its own optimistic inputs.

### Human-in-the-Loop
The workflow pauses to ask the user before spending tokens/money on the "Risk Factor Enrichment" (Massive.com API), showing respect for user resources.

### Cluster Processing
Grouping survivors by industry for the Analyst phase allows for comparative analysis (peers) within a single context window, which improves the quality of the "Moat" and "Valuation" assessments.

---

## 3. Weaknesses

### Reliance on LLM Arithmetic
The Analyst agent is asked to calculate CAGR and valuation formulas (`((Price Target / Current Price) ^ (1 / Years)) - 1`) purely via text generation. LLMs are notoriously prone to minor arithmetic errors without a code execution tool.

### Context Window Bottleneck (Phase 1)
The "Full NYSE" scan attempts to list all NYSE stocks. Even with filtering, passing thousands of tickers to the Screener agent in one prompt to "Process in batches" relies heavily on the model's ability to self-regulate iteration, which can be flaky.

### Parsing Fragility
The Orchestrator relies on reading Markdown tables generated by the Screener. If the Screener hallucinates the table format or adds conversational filler, the Orchestrator might fail to extract the survivor list correctly.

### Python-in-Bash Dependency
The `fmp-api.sh` script relies on `python3` being installed and available in the environment to parse JSON for SBC and Shares. If the environment lacks Python or the specific JSON structure changes, these specific calls will fail silently or break the pipe.

---

## 4. Potential Issues

### A. Performance & Token Economy
- **The "Full Scan" Trap:** Running `list-nyse` returns ~3,000+ active tickers. Pumping this entire list into the Screener's context window is inefficient and likely to hit token limits or degradation in reasoning. The Screener agent is instructed to "Process in batches," but the mechanism for doing this *within* the agent loop isn't strictly defined in code — it relies on the LLM's discretion.
- **Cache Invalidation:** The `screener` cache has a 7-day TTL. If a stock drops 20% today (entering the "Deep Value" territory), the scanner won't see it for a week unless the user forces `--fresh`.

### B. Accuracy / Logic
- **Date Hallucinations:** The prompts ask for "current" dates and specific timelines. Unless the LLM system prompt injects the current date, calculations regarding "Years to Target" or "YTD" might be inaccurate.
- **Valuation Formula Drift:** The Analyst is told: "Do NOT revise inputs to produce a better score." However, LLMs function probabilistically. If the first generated token of the score implies a low number, the model might "back-propagate" reasoning in the subsequent text to justify it, or vice versa.

### C. Security
- **API Key Handling:** The script sources `.env` files from multiple locations (Plugin root, User config, Data dir). While flexible, this increases the attack surface if the plugin is installed in a shared environment.
- **Arbitrary Code Execution (Theoretical):** The Orchestrator uses the `Task` tool. If a malicious prompt injection occurred via the `current-portfolio.md` or a manipulated web search result, it could theoretically trick the Orchestrator into executing bash commands outside the scope of the scanner, although Claude's sandbox mitigates this.

## 5. Suggested Improvements

### 1. Implement a Calculator Tool or Python Execution
Instead of asking the Analyst to "Calculate: Price Target...", provide a specific tool or instruct the agent to write and execute a Python script to perform the scoring math. This ensures the CAGR and Decision Score are mathematically precise.

### 2. Hard-Code the Batching Logic
Do not rely on the Screener agent to "process in batches." Modify the `scan-stocks` skill or Orchestrator to:
1. Fetch the NYSE list via `fmp-api.sh`.
2. Split the list into chunks of 50-100 tickers via Bash.
3. Feed these chunks to the Screener agent iteratively.

This ensures no survivors are missed due to context window overcrowding.

### 3. Structured Output (JSON)
Modify the Screener and Analyst output instructions to require a JSON block at the end of their response.
- *Current:* Markdown Table.
- *Proposed:*
  ```json
  {
    "survivors": [
      {"ticker": "CPS", "cluster": "Auto Parts", "flags": ["solvency_borderline"]}
    ]
  }
  ```
This makes the Orchestrator's job of parsing survivors deterministic and robust.

### 4. Robust Python Parsing in Bash
In `fmp-api.sh`, replace the inline Python one-liners with `jq` if available, or a more robust Python "Here Document" script that handles missing keys/exceptions gracefully.
- *Issue:* `row.get('stockBasedCompensation',0)` handles missing keys, but if the JSON structure shifts (e.g., nested inside another 'data' object), the script breaks.

### 5. Add "Freshness" to Critical Checks
For the **Step 1 Filter (Broken Chart Check)**, ensure `fmp-api.sh price-history` uses a shorter TTL (e.g., 1 day) or force a fresh fetch. A stock down 60% is a volatile situation; 7-day old price data is too stale for a screener.

> **Note:** Price-history already has 1-day TTL — this is correctly handled. The concern is more about the `screener` endpoint (7-day TTL) which may miss newly qualifying stocks.

### 6. Portfolio Integration
The `current-portfolio.md` is a static file. Consider adding a command to the skill (`/update-portfolio`) that parses a user-pasted CSV or connects to a broker API to keep this file current, as the "Portfolio Impact" section is critical for the final decision.

### Strategy Pipeline (Steps 1-8)

| Step | Agent | What It Does |
|------|-------|-------------|
| 1 | Screener | Find ideas: broken chart (60%+ off ATH), exclude industries, secular decline check, quality sniff, double-plus potential |
| 2 | Screener | 5 checks: solvency, dilution, revenue growth, ROIC, valuation |
| 3 | Analyst | Competitor comparison within industry cluster |
| 4 | Analyst | Qualitative deep dive: moats, management, issues, compensation, catalysts |
| 5 | Analyst | 4-input valuation model (Revenue x FCF Margin x Multiple / Shares) |
| 6 | Analyst | Decision scoring with position sizing |
| 7 | (Manual) | Portfolio management / monitoring (not automated) |
| 8 | (Manual) | Sell rules (not automated) |

### Hard Rules (Non-Negotiable)

| Rule | Enforcement |
|------|-------------|
| 30% CAGR hurdle (base case) | Analyst Step 5 |
| No catalysts = automatic pass | Analyst Step 4 Q5 |
| Excluded industries | Screener Step 1 |
| Max 12 positions | Orchestrator Step 5 |
| 50% single-theme cap | Orchestrator Step 5 |
| Score < 45 = no investment | Scoring formula |
| Scoring runs once | Analyst Step 6 discipline |
| SBC-to-pay-debt = immediate DQ | Screener Check 2 |
